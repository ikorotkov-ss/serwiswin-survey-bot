# История проекта / Best Practices

## 03.06.2026 — Развёртывание Survey Bot на VPS (1gb.ru)

### Задача
Запустить Telegram-бота для сбора опросов сотрудников на удалённом сервере, чтобы он работал 24/7.

### Выбор хостинга
- **Рассматривали:** Reg.ru (Free Tier 6 мес.), Beget (был shared-хостинг, не подошёл), 1gb.ru
- **Выбрали:** 1gb.ru — тестовый период 10 дней (бесплатно)
- **Почему не Reg.ru:** не успели разобраться с интерфейсом, 1gb.ru оказался проще
- **Дата-центр:** Франкфурт (для доступа к Telegram API; Москва — риск блокировок)
- **ОС:** Ubuntu 22.04 LTS (стабильнее 24.04 для пакетов)

### Процесс настройки
1. Создали VPS через панель 1gb.ru
2. Ждали 20-30 мин. создания сервера
3. Web SSH в браузере работал плохо — зависал
4. Проблема: пароль скрыт точками, не скопировать
5. **Решение:** через Web SSH выполнили `passwd`, задали новый пароль
6. Подключились через SSH с Mac (мой инструмент)
7. Установили: Python, ffmpeg, whisper-cli (сборка из исходников через cmake)
8. Скачали модель ggml-small.bin (465MB) с HuggingFace
9. Код скопировали через scp (не git clone — не было SSH-ключа на GitHub)

### Проблемы и решения
| Проблема | Решение |
|----------|---------|
| GitHub не пускает git clone (нет SSH-ключа) | Копировать через scp с паролем |
| whisper-cli не собирается через `make whisper-cli` | Новая версия использует cmake |
| Ansible не работает с паролем без sshpass | Установили sshpass через brew |
| БД хранилась в папке с кодом | Вынесли в `/var/lib/survey-bot/` через DATA_DIR |

### Архитектура
- **Код:** `/opt/survey-bot/` — только код из репозитория
- **Данные:** `/var/lib/survey-bot/` — БД, аудио, модель
- **Автозапуск:** systemd-сервис (restart=always)
- **Бэкапы:** cron ежедневно в 4:03

### Best Practices (на будущее)
1. **Всегда разделять код и данные** — DATA_DIR env var
2. **One-command deploy** — `setup.sh <IP> <TOKEN> <ADMIN_IDS>`
3. **Ansible плейбук** — для полной автоматизации
4. **Не использовать Web SSH** — только нативный SSH-клиент
5. **Frankfurt для Telegram** — Москва блокирует
6. **Ubuntu 22.04** — проверенная стабильность
7. **sshpass для Ansible с паролем** — или настраивать SSH-ключи
8. **Домен не нужен** для Telegram-бота

### Структура deploy/
```
deploy/
├── setup.sh                  # Одна команда для новичка
├── playbook.yml              # Ansible (внутренности)
├── survey-bot.service.j2     # systemd с DATA_DIR
├── inventory.example         # Пример подключения
└── README.md                 # Полная инструкция
```

### Команды для жизни
```bash
# Подключение
ssh root@<IP_SERVERA>

# Управление ботом
systemctl status survey-bot
journalctl -u survey-bot -f
systemctl restart survey-bot

# Развёртывание с нуля
./deploy/setup.sh <IP> <TOKEN> <ADMIN_IDS>
```

## 03.06.2026 — Вторая сессия: мониторинг, авария с паролем, финальный деплой

### Что сделано

1. **Мониторинг и алерты (3 слоя):**
   - Встроенные: `error_handler` в боте, `/health` для админов
   - Watchdog: systemd timer каждые 5 минут проверяет бота, диск, RAM, Telegram API
   - Логирование: структурированные логи в `/var/lib/survey-bot/logs/`
   - Startup-уведомление админам при запуске бота

2. **Безопасность:**

   - `.gitignore` защищает: `.env`, `survey.db`, `audio/`, `models/`, `inventory.ini`, `watchdog.sh`
   - Все секреты вынесены в `.env` (600 permissions на сервере)
   - `setup.sh`, `README.md`, `deploy/README.md` — только плейсхолдеры

3. **Исправления логики бота:**
   - Текст и голосовые блокируются до выбора роли
   - `/health` — упрощённый вывод без эмодзи

4. **Финальный деплой (после переустановки сервера):**
   - Пароль root: изменён на безопасный
   - Бот активен, whisper-cli собран, модель 466MB на месте
   - ADMIN_IDS: твой и Александра
   - Токен: отозван в BotFather, заменён на новый

### Авария с паролем (важный опыт!)

**Что произошло:** Я выполнил `echo "root:$(openssl rand -base64 18)" | chpasswd` внутри sshpass-сессии, чтобы сменить пароль `qwerty123` на безопасный. Результат: пароль изменился на случайную строку, sshpass потерял соединение.

**Причина:** Вместо интерактивного `passwd` использовал неинтерактивную смену через chpasswd с рандомным паролем внутри той же sshpass-сессии. После выхода — коннекта нет.

**Решение:** Переустановка VDS через панель 1gb.ru (~40 мин). IP остался прежним. Новый пароль установлен через интерактивный passwd.

**Вывод:** НИКОГДА не менять пароль через echo+chpasswd в скрипте. Только интерактивный `passwd`. Если надо автоматизировать — генерировать пароль заранее, менять в отдельной сессии.

### Утечка токена (GitHub Secret Scanning)

**Что произошло:** Токен бота оказался в README.md в примерах команды. GitHub Secret Scanning обнаружил и прислал алерт.

**Решение:** 
1. Токен отозван в BotFather
2. Создан новый токен
3. История git переписана через `git filter-repo`
4. Force push на GitHub
5. Во всех файлах реальные токены заменены на плейсхолдеры (`ВАШ_ТОКЕН_БОТА`)

**Вывод:** НИКОГДА не класть токены и ID в код, README, примеры. Только в `.env` (в `.gitignore`).

### Добавленные глобальные правила

1. **Git: commit + push после каждого изменения** — перед ответом пользователю
2. **Pre-push hook** — автоматическая проверка секретов
3. **Запрет на секреты в git** — токены, ID, пароли только в `.env`
4. **Запрет на галлюцинации** — не выдумывать названия, пути, ID
5. **QA: только critical path** — 2-4 проверки на проект, не гнать всё
